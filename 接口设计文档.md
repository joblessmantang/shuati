## 一、接口设计概述

### 1.1 整体架构

本系统采用典型的**前后端分离**架构，前端 Vue 3 通过 axios 与后端 Node.js/Express 通信。后端遵循 **RESTful** 设计风格，所有接口统一前缀为 `/api`，响应格式为 JSON。

**前后端通信约定：**

- 后端统一响应包装：`{ success: true, data: ... }` 或 `{ success: false, message: "..." }`
- 前端 axios 响应拦截器自动解包 `payload.data`，遇 401 自动跳转登录页
- 所有涉及用户资源的接口通过 `userId` 参数区分（GET `?userId=xxx`，POST body `userId`）

### 1.2 技术选型依据

| 层次 | 技术 | 选型理由 |
|------|------|----------|
| 前端框架 | Vue 3 (Composition API) | 渐进式框架，学习成本低，生态丰富，适合交互密集型刷题场景 |
| HTTP 客户端 | axios | 支持请求/响应拦截，统一 Token 注入，统一错误处理 |
| 后端框架 | Express.js | 轻量、灵活，中间件机制成熟，Node.js 生态完善 |
| 数据库 | MySQL | 结构化数据存储，支持复杂联表查询，适合关系型刷题数据 |
| ORM | mysql2/promise（原生 SQL + 连接池） | 性能优于 ORM，可精确控制 SQL，适合教学场景的复杂聚合查询 |
| 认证 | JWT (Bearer Token) | 无状态认证，适合分布式前端，支持 token 过期机制 |
| 文件上传 | multer | Express 官方推荐，处理 multipart/form-data，支持文件类型/大小限制 |

## 三、核心接口设计详解

### 3.1 认证接口设计

#### 登录接口 `POST /api/auth/login`

**请求体：**
```json
{
  "username": "tl",
  "password": "123456"
}
```

**响应（成功）：**
```json
{
  "success": true,
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "user": {
      "id": 1,
      "username": "tl",
      "role": "admin",
      "avatar": null
    }
  }
}
```

**设计要点：**
- 使用 `bcryptjs` 对密码进行 hash 存储（不存明文）
- 使用 `jsonwebtoken` 生成带过期时间的 token
- Token 中仅包含 `id` 和 `username`，角色信息从数据库实时查询（防止 token 伪造角色）
- 所有需要认证的接口通过 `authMiddleware` 中间件统一校验

#### JWT 中间件认证流程

```
请求 → authMiddleware
  │
  ├─ 无 Authorization Header → 401 未提供令牌
  ├─ Token 格式错误 → 401 无效令牌
  ├─ Token 过期 → 401 令牌已过期
  ├─ 用户不存在 → 401 用户不存在
  └─ 验证通过 → next() → 控制器
```

```javascript
// houduan/src/middlewares/auth.js
const jwtMiddleware = async (req, res, next) => {
    const authHeader = req.headers.authorization;
    if (!authHeader || !authHeader.startsWith('Bearer ')) {
        return res.status(401).json({ success: false, message: '未提供认证令牌' });
    }
    const token = authHeader.split(' ')[1];
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    const [users] = await pool.execute('SELECT id, username, role FROM users WHERE id = ?', [decoded.id]);
    if (users.length === 0) {
        return res.status(401).json({ success: false, message: '用户不存在' });
    }
    req.user = { id: decoded.id, username: decoded.username, role: users[0].role };
    next();
};
```

### 3.2 知识图谱接口设计

#### 能力分析接口 `GET /api/analysis/ability`

**查询参数：** `?userId=xxx`

**响应：**
```json
{
  "success": true,
  "data": [
    {
      "categoryId": 1,
      "categoryName": "JavaScript",
      "total": 25,
      "correct": 18,
      "ability": 72,
      "isWeak": true
    },
    {
      "categoryId": 2,
      "categoryName": "CSS",
      "total": 0,
      "correct": 0,
      "ability": null,
      "isWeak": false
    }
  ]
}
```

**设计要点：**

1. **纯聚合计算，不建新表**：能力数据来自 `user_answers`（每题答题记录）、`questions`（题目分类）、`categories`（分类信息）三表联查
2. **能力值算法**：`ability = (correct / total) * 100`，无做题记录返回 `null`
3. **薄弱判定**：来自 `wrong_book` 表，统计错题本中各分类的错题数量，取 TOP3 标记为薄弱
4. **多源并行查询**：使用 `Promise.all` 并行执行独立查询，提升响应速度

```javascript
// 核心 SQL：分类能力聚合
SELECT
    q.categoryId,
    COUNT(ua.id) AS total,
    SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct
FROM user_answers ua
JOIN questions q ON ua.question_id = q.id
WHERE ua.user_id = ? AND q.categoryId IS NOT NULL
GROUP BY q.categoryId
```

### 3.3 个性化推荐接口设计

#### 首页推荐接口 `GET /api/analysis/recommendations`

**查询参数：** `?userId=xxx`

**响应：**
```json
{
  "success": true,
  "data": {
    "weakCategories": [
      { "categoryId": 3, "categoryName": "HTML", "reason": "错题本中有 5 道相关题目" }
    ],
    "recommendMode": {
      "mode": "errors",
      "reason": "错题较多，建议先专攻错题本"
    },
    "wrongRecommendations": [
      { "id": 42, "title": "...", "reason": "错题本 · 需要复习", "type": "wrong" }
    ],
    "favoriteRecommendations": [
      { "id": 15, "title": "...", "reason": "收藏题目", "type": "favorite" }
    ]
  }
}
```

**设计要点（规则推荐，无推荐引擎）：**

1. **薄弱分类推荐**：联查 `wrong_book` 和 `user_answers`，错题最多 + 正确率最低的分类优先
2. **练习模式推荐**：基于用户最近 5 次练习正确率均值决策：
   - `avgAcc >= 80%` → 推荐 `challenge`（计时挑战）
   - `avgAcc >= 50%` → 推荐 `mix`（混合练习）
   - `avgAcc < 50%` → 推荐 `errors`（错题专攻）
3. **近期错题/收藏推荐**：取最近 3 条，按时间降序

### 3.4 每日签到接口设计

#### 获取签到记录 `GET /api/checkIn`

**查询参数：** `?userId=xxx`

**响应：**
```json
{
  "success": true,
  "data": {
    "lastCheckInDate": "2026-04-23",
    "continuousDays": 4,
    "totalDays": 12,
    "weekCheckInDates": ["2026-04-21", "2026-04-22", "2026-04-23"],
    "monthCheckInDates": ["2026-04-01", "2026-04-15", "2026-04-21", "2026-04-22", "2026-04-23"]
  }
}
```

**设计要点：**

1. **连续签到算法**：从今天往前数，依次检查每一天是否有签到记录，如果日期连续则累加，中断则停止
2. **防重签机制**：数据库 `checkin_date` 字段设置 `UNIQUE` 约束，后端再次校验，双重保护
3. **时间范围计算**：前端展示需要本周（周一~周日）和本月（1日~月末）签到日期列表，后端一次性返回

#### 执行签到 `POST /api/checkIn`

**请求体：** `{ "userId": 1 }`

**连续天数计算逻辑：**

```javascript
// 判断昨天是否有记录 → 有则连续，无则重新开始
const isContinuous = yesterdayRows.length > 0;
const newStreak = isContinuous ? yesterdayStreak + 1 : 1;
const newTotal = yesterdayTotal + 1;
```

### 3.5 学习资料接口设计

#### 推荐资料接口 `GET /api/resources/recommended`

**查询参数：** `?userId=xxx`

**设计要点（规则推荐）：**

1. **薄弱点匹配**：取用户最近 30 天内正确率 < 60% 的分类
2. **多表关联**：`resources` ↔ `resource_topics` ↔ `categories`，匹配薄弱分类
3. **兜底策略**：无薄弱数据时，返回平台标记推荐的资料（`is_recommended = 1`）

### 3.6 文件上传接口设计

#### 封面上传 `POST /api/upload/cover`

**请求格式：** `multipart/form-data`，字段名 `cover`

**设计要点：**

1. **multer 中间件**：处理文件流，支持 `diskStorage` 磁盘存储
2. **双重格式校验**：扩展名白名单（`/jpeg|jpg|png|webp|gif/`）+ MIME 类型校验
3. **文件大小限制**：封面 5MB，资料文件 100MB
4. **文件重命名**：`cover_${Date.now()}.${ext}`，避免文件名冲突
5. **静态文件服务**：`app.use('/uploads', express.static(...))`，前端直接访问

## 五、安全性设计

### 5.1 认证与授权

| 机制 | 实现 |
|------|------|
| 密码存储 | bcryptjs hash，不可逆 |
| 通信安全 | HTTPS（生产环境） |
| Token 管理 | JWT，存储于 localStorage，设置过期时间 |
| 管理员操作 | `authMiddleware` 校验角色为 `admin` |
| SQL 注入防护 | 参数化查询（`pool.execute` / `pool.query` 占位符） |

### 5.2 错误处理

所有接口统一的错误响应格式：

```json
{
  "success": false,
  "message": "错误描述信息"
}
```

全局错误中间件 `errorHandler` 捕获所有未处理的异常，返回 500 并记录日志。
