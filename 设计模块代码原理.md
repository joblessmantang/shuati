## 一、整体架构设计

### 1.1 分层架构

系统采用经典的三层架构（Controller - Service - Data），每层职责清晰：

```
┌─────────────────────────────────────────────────┐
│              Controller 层（控制器）               │
│  - 接收 HTTP 请求，解析参数                       │
│  - 调用 Service 层                               │
│  - 处理响应格式，统一返回 { success, data }       │
└─────────────────────┬───────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────┐
│              Service 层（业务逻辑层）               │
│  - 核心业务逻辑处理                               │
│  - 数据聚合与转换                                │
│  - 事务编排（多个 DB 操作）                        │
└─────────────────────┬───────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────┐
│              Data 层（数据库）                    │
│  - mysql2/promise 连接池                        │
│  - 参数化 SQL 查询                               │
│  - 结果集处理                                    │
└─────────────────────────────────────────────────┘
```

**职责分离的优势：**
- Controller 只负责请求/响应，不含业务逻辑，便于单元测试
- Service 包含纯业务逻辑，可跨 Controller 复用
- 数据层统一数据库连接管理，连接池复用减少开销

### 1.2 前端状态管理架构（Pinia）

前端采用 Pinia（Vue 3 官方推荐状态管理库），每个功能模块独立 store：

| Store | 职责 |
|-------|------|
| `user.js` | 用户登录状态、用户信息 |
| `learning.js` | 签到状态、学习目标、专题列表 |
| `learning.js` | 答题状态、计时器（通过 Quiz.vue 组件内部管理） |

## 二、数据库连接池设计

### 2.1 连接池原理

```javascript
// houduan/src/config/database.js
const pool = mysql.createPool({
    host: process.env.DB_HOST || 'localhost',
    port: process.env.DB_PORT || 3306,
    user: process.env.DB_USER || 'root',
    password: process.env.DB_PASSWORD || '',
    database: process.env.DB_NAME || 'interview_platform',
    waitForConnections: true,
    connectionLimit: 10,    // 最多 10 个并发连接
    queueLimit: 0,          // 队列无限制
    charset: 'utf8mb4'
});
```

**设计要点：**

1. **连接复用**：连接池维护 10 个持久连接，请求到来时从池中取空闲连接，用完归还，避免频繁建立/断开 TCP 连接的开销
2. **`charset: 'utf8mb4`**：支持 emoji 和特殊字符存储（如题目选项中的特殊符号）
3. **环境变量配置**：数据库连接信息通过 `.env` 文件管理，开发/生产环境切换时无需修改代码
4. **启动时连接测试**：`startServer()` 调用 `testConnection()`，数据库不可用时阻止服务启动

## 三、知识图谱聚合计算设计

### 3.1 需求分析

知识图谱需要展示用户在各个知识分类上的能力掌握程度，核心挑战是：**如何从分散的答题记录中高效聚合出分类级别的能力数据？**

### 3.2 纯聚合计算策略（无新表）

传统方案会新建一张 `user_category_ability` 表，每次答题后更新能力值。本系统采用**纯实时聚合**，完全不建新表：

```javascript
// houduan/src/services/analysisService.js
async function getCategoryAbility(userId) {
    // 1. 获取所有分类
    const [cats] = await pool.execute('SELECT id, name FROM categories ORDER BY id ASC');

    // 2. 从 user_answers 实时聚合各分类的正确率
    const [rows] = await pool.query(`
        SELECT
            q.categoryId,
            COUNT(ua.id)                          AS total,
            SUM(CASE WHEN ua.is_correct = 1 THEN 1 ELSE 0 END) AS correct
        FROM user_answers ua
        JOIN questions q ON ua.question_id = q.id
        WHERE ua.user_id = ? AND q.categoryId IS NOT NULL
        GROUP BY q.categoryId
    `, [userId]);

    // 3. 聚合结果映射
    const statMap = {};
    rows.forEach(r => {
        statMap[r.categoryId] = {
            total: Number(r.total) || 0,
            correct: Number(r.correct) || 0
        };
    });

    // 4. 获取薄弱分类（错题本中题目最多的分类 TOP3）
    const [weakRows] = await pool.query(`
        SELECT q.categoryId, COUNT(*) as cnt
        FROM wrong_book wb
        JOIN questions q ON wb.questionId = q.id
        WHERE wb.userId = ? AND q.categoryId IS NOT NULL
        GROUP BY q.categoryId
        ORDER BY cnt DESC
        LIMIT 3
    `, [userId]);
    const weakCatIds = new Set(weakRows.map(r => r.categoryId));

    // 5. 构建返回数据
    return cats.map(cat => {
        const stat = statMap[cat.id] || { total: 0, correct: 0 };
        const ability = stat.total > 0
            ? Math.round((stat.correct / stat.total) * 100)
            : null;
        return {
            categoryId: cat.id,
            categoryName: cat.name,
            total: stat.total,
            correct: stat.correct,
            ability,       // null 表示未练习过
            isWeak: weakCatIds.has(cat.id)
        };
    });
}
```

**优势分析：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 预计算表（新建表） | 查询快 | 数据同步复杂，可能不一致；多占用存储 |
| 纯实时聚合（本方案） | 数据永远一致；无需额外存储 | 每次请求需要聚合计算（但数据量有限，毫秒级） |

考虑到用户答题记录数量有限（通常数百~数千条），实时聚合的计算量可忽略不计，而数据一致性优势显著。

## 四、规则推荐引擎设计

### 4.1 推荐策略架构

本系统的推荐系统基于**规则引擎**（而非机器学习推荐引擎），通过预定义的规则集匹配用户行为数据，生成个性化推荐。

```javascript
// 核心推荐策略（来自 analysisService.js）
async function getHomeRecommendations(userId) {
    // 并行获取多源数据
    const [weakCats, recentHistory, recentWrong, recentFavorites] = await Promise.all([
        getWeakCategoryRecommendations(userId),    // 薄弱分类
        pool.query(`SELECT ... FROM practice_history ... LIMIT 10`),  // 历史
        pool.query(`SELECT ... FROM wrong_book ... ORDER BY created_at DESC LIMIT 3`), // 近期错题
        pool.query(`SELECT ... FROM favorites ... ORDER BY created_at DESC LIMIT 3`)  // 收藏
    ]);

    // 推荐练习模式
    const recommendMode = analyzePracticeMode(recentHistory);

    return {
        weakCategories: weakCats,
        recommendMode,
        wrongRecommendations,
        favoriteRecommendations
    };
}

// 练习模式决策树
function analyzePracticeMode(history) {
    if (!history || history.length === 0) {
        return { mode: 'normal', reason: '开始你的第一次练习吧！' };
    }

    const recentAccs = history.slice(0, 5).map(h =>
        h.total > 0 ? (h.score / h.total) * 100 : 0
    );
    const avgAcc = recentAccs.reduce((a, b) => a + b, 0) / recentAccs.length;

    if (avgAcc >= 80) return { mode: 'challenge', reason: '正确率优秀，试试计时挑战模式！' };
    if (avgAcc >= 50) return { mode: 'mix', reason: '建议混合练习，查漏补缺' };
    return { mode: 'errors', reason: '错题较多，建议先专攻错题本' };
}
```

**推荐策略规则表：**

| 规则 ID | 触发条件 | 推荐结果 | 优先级 |
|---------|---------|---------|--------|
| R-01 | 近期正确率 >= 80% | 推荐「计时挑战」模式 | 1 |
| R-02 | 近期正确率 50-80% | 推荐「混合练习」模式 | 2 |
| R-03 | 近期正确率 < 50% | 推荐「错题练习」模式 | 3 |
| R-04 | 无练习记录 | 推荐「普通练习」模式 | 4 |
| R-05 | 错题本有题 | 推荐对应分类的近期错题 | 辅助 |
| R-06 | 收藏中有题 | 推荐近期收藏的题目 | 辅助 |

**规则引擎的可扩展性：** 未来可替换为基于协同过滤或深度学习的推荐引擎，只需实现 `getHomeRecommendations` 接口契约，前端无需修改。

## 五、连续签到算法设计

### 5.1 问题定义

连续签到天数的计算面临边界情况：用户可能在某天忘记签到导致连续中断，或者跨月/跨年签到。

### 5.2 算法实现

```javascript
// 签到服务中的连续天数计算
async calculateStreak(userId) {
    const [rows] = await pool.query(`
        SELECT checkin_date, streak_days
        FROM check_ins
        WHERE user_id = ?
        ORDER BY checkin_date DESC   -- 按日期降序
    `, [userId]);

    if (rows.length === 0) return 0;

    let streak = 0;
    let expectedDate = new Date();

    for (const row of rows) {
        const d = new Date(row.checkin_date);
        const today = new Date();
        today.setHours(0, 0, 0, 0);
        const checkDate = new Date(d);
        checkDate.setHours(0, 0, 0, 0);

        // 计算日期差（天数）
        const diff = Math.floor((today - checkDate) / (1000 * 60 * 60 * 24));

        // 如果当前记录的日期等于预期日期（连续），streak + 1
        if (diff === streak) {
            streak++;
            expectedDate.setDate(expectedDate.getDate() - 1);
        } else {
            break;  // 不连续，停止
        }
    }
    return streak;
}
```

**算法说明：**

- 从今天往前遍历签到记录，检查每条记录是否在预期日期上
- `streak` 既作为计数器，也作为"预期回溯天数"的偏移量
- 遇到中断（`diff !== streak`）立即停止，保证连续性判定正确
- 时间复杂度 O(n)，n 为签到总次数

**执行签到时的连续天数计算（更简单的方式）：**

```javascript
// 检查昨天是否有记录，决定是否连续
const isContinuous = yesterdayRows.length > 0;
const newStreak = isContinuous ? yesterdayStreak + 1 : 1;
```

在签到时，不需要遍历，只需查昨天是否有记录即可（O(1)），而 `calculateStreak` 用于历史校验场景。

## 六、学习资料推荐算法

### 6.1 多对多关联设计

学习资料（`resources`）与知识点分类（`categories`）是多对多关系，通过中间表 `resource_topics` 实现：

```
resources ────── resource_topics ────── categories
   id                  resource_id          id
   title               category_id          name
   author
   ...
```

```javascript
// 资料推荐查询（resourceService.js）
async getRecommendedForUser(userId, limit = 4) {
    // 1. 获取用户最近 30 天内正确率 < 60% 的分类
    const [weakRows] = await pool.query(`
        SELECT c.id as category_id, c.name as category_name,
               COUNT(*) as total, ROUND(SUM(ua.is_correct) / COUNT(*) * 100) as accuracy
        FROM user_answers ua
        JOIN questions q ON ua.question_id = q.id
        JOIN categories c ON q.categoryId = c.id
        WHERE ua.user_id = ? AND ua.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
        GROUP BY c.id
        HAVING accuracy < 60
        ORDER BY total DESC
        LIMIT 5
    `, [userId]);

    if (weakRows.length === 0) {
        // 2a. 无薄弱点 → 返回平台推荐资料
        return { items: recommendedItems, reason: '精选推荐', weakCategories: [] };
    }

    // 2b. 有薄弱点 → 匹配关联知识点
    const weakCategoryIds = weakRows.map(w => w.category_id);
    const [rows] = await pool.query(`
        SELECT DISTINCT r.* ...
        FROM resources r
        JOIN resource_topics rt ON rt.resource_id = r.id
        WHERE rt.category_id IN (?)
        ORDER BY r.is_recommended DESC, r.view_count DESC
        LIMIT ?
    `, [weakCategoryIds, limit]);

    return {
        items: formattedItems,
        reason: `针对「${weakRows[0].category_name}」等薄弱分类推荐`,
        weakCategories: weakRows,
    };
}
```

**设计亮点：**

1. **规则匹配**：将用户薄弱知识点与资料的知识点标签关联，而非简单的"猜你喜欢"
2. **时间衰减**：只考察最近 30 天数据，避免用远古记录判定当前薄弱点
3. **兜底机制**：新用户无历史数据时，回退到平台标记推荐的资料
4. **IN 查询优化**：`category_id IN (?)` 配合 `JOIN` 实现多分类联合匹配

## 七、前端 axios 拦截器设计

### 7.1 请求/响应拦截器架构

```javascript
// shuati/src/api/index.js
const request = axios.create({
  baseURL: process.env.VUE_APP_API_BASE || "http://localhost:3000/api",
  headers: { "Content-Type": "application/json" },
});

// 请求拦截：自动注入 JWT Token
request.interceptors.request.use((config) => {
  const token = localStorage.getItem("token");
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// 响应拦截：统一解包 + 401 处理
request.interceptors.response.use(
  (res) => {
    const payload = res.data;
    // 统一解包：后端 { success, data } → 前端直接拿到 data
    if (payload && payload.success !== undefined && payload.data !== undefined) {
      return { ...res, data: payload.data };
    }
    return res;
  },
  (err) => {
    if (err.response && err.response.status === 401) {
      localStorage.removeItem("token");
      localStorage.removeItem("user");
      // 清除状态后跳转到登录页
      if (window.location.pathname !== "/login") {
        window.location.href = "/#/login";
      }
    }
    return Promise.reject(err);
  }
);
```

**设计要点：**

1. **Token 自动注入**：所有请求自动携带 `Authorization: Bearer <token>`，无需每个 API 调用手动添加
2. **统一响应解包**：前端代码直接使用 `const { data } = await api.list()` 即可拿到业务数据，无需关心后端包装格式
3. **401 全局处理**：任何接口返回 401，自动清除登录状态并跳转登录页，确保安全
4. **Promise 链式传递**：错误通过 `Promise.reject(err)` 传递，调用方可用 `try/catch` 统一处理

### 7.2 API 模块化设计

```javascript
// 每个功能域一个 API 对象，职责单一
export const authApi    = { login, register, me, ... };
export const questionApi = { list, search, get, create, ... };
export const checkInApi  = { getRecord, checkIn };
export const goalApi     = { getGoal, setGoal, addProgress, ... };
export const analysisApi = { getAbility, getRecommendations };
```

**优势：** 调用方只需导入需要的 API 对象，按需加载（Tree-shaking），且各模块 API 变更互不影响。

## 八、Pinia 状态管理设计

### 8.1 学习激励状态管理（learning.js）

```javascript
// shuati/src/stores/learning.js
export const useLearningStore = defineStore('learning', {
  state: () => ({
    checkInRecord: null,
    currentGoal: null,
    topics: [],
    // ...
  }),

  getters: {
    // 计算属性：今日是否已签到
    isCheckedInToday: (state) => {
      if (!state.checkInRecord?.lastCheckInDate) return false;
      const today = new Date().toISOString().slice(0, 10);
      return state.checkInRecord.lastCheckInDate === today;
    },
    // 计算属性：目标进度百分比
    goalProgressPercent: (state) => {
      const t = state.currentGoal?.target ?? 0;
      if (t === 0) return 0;
      return Math.min(Math.round(((state.currentGoal?.current ?? 0) / t) * 100), 100);
    },
    // 计算属性：目标达成文案（动态文案生成）
    goalLabel: (state) => {
      if (!state.currentGoal) return '设定学习目标';
      if (state.currentGoal.current >= state.currentGoal.target) return '今日目标已完成！';
      const rem = state.currentGoal.target - state.currentGoal.current;
      return state.currentGoal.type === 'daily' ? `今日还差 ${rem} 题` : `本周还差 ${rem} 题`;
    }
  },

  actions: {
    // 异步操作：签到
    async doCheckIn(userId) {
      if (this.isCheckingIn) return;
      this.isCheckingIn = true;
      try {
        const { data } = await checkInApi.checkIn(userId);
        this.checkInRecord = data;  // 自动更新响应式状态
      } finally {
        this.isCheckingIn = false;
      }
    },

    // 异步操作：追加目标进度（答题提交时调用）
    async addToGoalProgress(questionCount) {
      if (!this.currentGoal?.id) return;
      if (typeof questionCount !== 'number' || questionCount <= 0) return;  // 参数校验
      // 先更新本地状态（乐观更新）
      this.currentGoal = { ...this.currentGoal, current: this.currentGoal.current + questionCount };
      try {
        await goalApi.addProgress(this.currentGoal.id, questionCount);
      } catch {
        // 同步失败时降级：本地仍更新（前端优先体验）
      }
    }
  }
});
```

**设计模式分析：**

1. **State（状态）**：唯一真实数据来源
2. **Getters（计算属性）**：基于 state 的派生数据，自动追踪依赖
3. **Actions（操作）**：修改 state 的唯一途径，支持异步操作

**乐观更新模式**：`addToGoalProgress` 中先更新本地状态，再异步同步后端，用户感知无延迟。后端同步失败时仅警告，不回滚本地状态（降级体验优先）。

## 九、练习模式策略设计

### 9.1 模式策略模式（Strategy Pattern）

五种练习模式通过策略模式实现，每种模式对应不同的抽题/计时逻辑：

```javascript
// Quiz.vue 中的模式路由
const modeHandlers = {
    normal:    () => fetchRandomQuestions(),
    errors:    () => fetchFromWrongBook(),
    mix:       () => fetchMixQuestions(),    // 50% 错题 + 50% 新题
    challenge: () => fetchChallengeQuestions(5),  // 固定 5 题
    topic:     () => fetchTopicQuestions(topicId),
    review:    () => fetchRandomQuestions()
};
```

**各模式行为对照：**

| 模式 | 抽题策略 | 计时 | 提交行为 |
|------|---------|------|---------|
| `normal` | 随机抽题 | 无 | 正常提交，弹出结果 |
| `errors` | 优先错题本，无则随机 | 无 | 正常提交 |
| `mix` | 错题+新题各半 | 无 | 正常提交 |
| `challenge` | 固定 5 题 | 1/2/3/5 分钟可选 | 超时 SweetAlert 2s 后自动交卷 |
| `review` | 随机抽题 | 无 | 提交后直接展示答案 |

## 十、文件上传架构设计

### 10.1 multer 多实例配置

```javascript
// houduan/src/middlewares/upload.js
// 两个独立的 multer 实例，配置不同
const uploadCover = multer({
    storage: coverStorage,    // 封面：保存到 uploads/covers/
    fileFilter: coverFilter, // 图片格式校验
    limits: { fileSize: 5 * 1024 * 1024 }  // 5MB
});

const uploadFile = multer({
    storage: fileStorage,     // 资料：保存到 uploads/files/
    fileFilter: fileFilter,   // PDF 格式校验
    limits: { fileSize: 100 * 1024 * 1024 }  // 100MB
});
```

**设计要点：**

1. **多实例分离**：不同文件类型使用独立的 multer 实例，互不影响
2. **磁盘存储（diskStorage）**：非内存存储，支持大文件（100MB），避免内存溢出
3. **时间戳命名**：`cover_${Date.now()}.${ext}` 保证文件名唯一性
4. **类型白名单**：正则校验扩展名 + MIME 类型，双重防护
5. **Express 静态服务**：`app.use('/uploads', express.static(...))`，前端直接访问，无需额外路由

## 十一、错误处理中间件设计

### 11.1 全局错误处理链

```javascript
// houduan/src/app.js
app.use(notFoundHandler);  // 404 处理（路由注册之后）
app.use(errorHandler);     // 全局异常捕获

// notFoundHandler: 处理请求到达了但没有匹配路由的情况
// errorHandler: 捕获所有 throw new Error() 和未处理的 promise rejection
```

**统一错误响应格式的优势：**
- 所有错误格式一致，前端只需一种错误处理逻辑
- 错误日志集中，便于排查
- 敏感信息（数据库错误详情）在生产环境被屏蔽
